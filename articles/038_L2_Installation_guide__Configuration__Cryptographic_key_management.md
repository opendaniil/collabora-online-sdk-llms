This section describes the private keys used by Collabora Online, where they are stored, and the procedures for their lifecycle and renewal. Collabora Online relies on the operating system and on the operating organisation’s existing PKI for key material. It stores no keys in a database or product-managed key store, integrates no hardware security module, and performs no automated key rotation. The private keys are flat files protected by operating-system file permissions and owned by the operating organisation, which provisions them and controls access through standard operating-system access controls.

### Private key inventory

| Key | Purpose | Location | Responsibility |
| --- | --- | --- | --- |
| TLS server private key | HTTPS termination | File at the configured ssl.key_file_path (default /etc/coolwsd/key.pem) | Operating organisation |
| Storage client private key (optional) | Mutual TLS to a WOPI or storage backend | File at the configured storage.ssl.key_file_path (unset by default) | Operating organisation |
| WOPI proof private key | Signs the X-WOPI-Proof header so that WOPI hosts can verify request authenticity | File at /etc/coolwsd/proof_key | Operating organisation |

The administration console additionally uses an ephemeral signing key for its session tokens. It is generated in memory when the process starts, is never written to disk or distributed, and is discarded when the process exits, so it requires no storage, backup, or renewal procedure.

### WOPI proof key generation

The WOPI proof key is generated automatically by the package postinstall step, so a standard installation always has one and most deployments never need to create it manually. It is not strictly required: if the file is absent, coolwsd still runs and simply omits the proof entry from its discovery document, so WOPI hosts that do not verify proof headers are unaffected. If it is missing and you do want proof support, it can be generated with:

```
sudo coolconfig generate-proof-key
```

or, if the configuration directory is not `/etc/coolwsd`, with ssh-keygen:

```
ssh-keygen -t rsa -N "" -m PEM -f "/etc/coolwsd/proof_key"
```

Unlike the TLS and storage keys, the proof key location is not set through a configuration option. It is fixed at the compile-time configuration directory (`/etc/coolwsd` in the standard packages), so the key is always at `/etc/coolwsd/proof_key`. When generated by `coolconfig`, the `proof_key` file is created with mode `0600` and owned by the `cool` service account. It must be readable by the coolwsd process. The public half of the key is published in the discovery document (`/hosting/discovery`) so that WOPI hosts can fetch it to verify the proof headers.

### Key lifecycle

Collabora Online loads each configured key file at startup and uses it for the lifetime of the process. It does not generate, rotate, back up, or expire the TLS server key, and it performs no automated rotation of the WOPI proof key. Key renewal is therefore a manual operational procedure that the operating organisation defines, schedules, and tracks within its own key-management policy. The product enforces no fixed renewal interval.

### Renewal procedures

 TLS server certificate and key

Renewed on the operating organisation’s own PKI cadence, for example the certificate authority’s validity period or ACME-based automation. To renew, install the new key and certificate files at the configured paths and restart the coolwsd service.

  WOPI proof key

To renew, remove `/etc/coolwsd/proof_key`, regenerate the key as shown above, and restart the coolwsd service. The new public key is then re-published in the discovery document, and WOPI hosts pick it up on their next discovery refresh. Proofs signed with the previous key are not honoured across the change, so renewal should be scheduled during a low-activity window.

### Backup and access control

The operating organisation is responsible for backing up the key files alongside the rest of the coolwsd configuration directory, in line with its backup policy. Loss of the proof key is recoverable by regeneration followed by a discovery refresh. Access to the key files is governed by operating-system file permissions and the `cool` service-account boundary. The operating organisation is responsible for restricting filesystem and administrative access to the host accordingly.
